<objective>
Close the two builder-side byte-fidelity defects (CR-01, CR-02) found by phase
verification (`02-VERIFICATION.md`, criterion 3 PARTIAL). The builder's
`end_tree` produces column lengths that diverge from upstream Treelite's
`AllocNode` invariant in two ways:

- **CR-01:** five per-node columns (`category_list_right_child`,
  `leaf_vector_begin`, `leaf_vector_end`, `category_list_begin`,
  `category_list_end`) are left empty (length 0) where upstream pushes one entry
  per node (length num_nodes).
- **CR-02:** six stat columns (`data_count`/`sum_hess`/`gain` + their `_present`
  companions) are unconditionally emitted at length num_nodes (all-false/zero)
  even when no node ever called the setter — upstream keeps them empty
  (length 0) unless at least one node set the stat.

Both are length-of-column divergences in the serialized v5 byte image, so a
treelite-rs-built model is not byte-identical to the upstream golden.

Purpose: A treelite-rs-built model must serialize to a byte image consistent
with upstream's column-length invariants (the project's core serialization
contract; SER-01 byte-fidelity for the builder path).

Output: corrected `end_tree` in `crates/treelite-builder/src/lib.rs` and a new
regression test `crates/treelite-builder/tests/column_fidelity.rs`.

OUT OF SCOPE — do NOT touch: DEF-02-01 (the XGBoost loader leaving
`sum_hess`/`gain`/CSR columns empty, leaf `split_index=-1`, `attributes={}`) is
owned by Phase 3. This plan fixes ONLY the builder's column-emission invariants
in `end_tree`. Do not modify any loader crate.
</objective>

    CR-01 (populate per-node columns, ported from upstream `AllocNode`): set the
    five currently-empty per-node columns to length num_nodes with upstream's

      - `category_list_right_child` = a vec of `false` repeated num_nodes times.
      - `leaf_vector_begin` = a vec of `0u64` repeated num_nodes times.
      - `leaf_vector_end` = a vec of `0u64` repeated num_nodes times.

      - `category_list_end` = a vec of `0u64` repeated num_nodes times.

    populates the `leaf_vector` or `category_list` value buffers in Phase 2, so

    `TreeBuf::from_owned(...)`. Leave the `leaf_vector` and `category_list` VALUE
    buffers as `TreeBuf::empty()` (unchanged — upstream leaves them empty too for
    these trees).

    CR-02 (empty-unless-set stat columns, ported from upstream `AllocNode`'s
    `if (!*_present_.Empty())` guards): before building each stat pair, compute
    whether ANY staged node set that specific stat, using the per-node
    `NodeStaging` present flags already tracked by the setters:
      - `any_data_count` = any node's `data_count_present` is true.
      - `any_sum_hess` = any node's `sum_hess_present` is true.
      - `any_gain` = any node's `gain_present` is true.
    For each of the three stat pairs INDEPENDENTLY:
      - If the corresponding `any_*` is false: assign BOTH the value column and the
        `_present` column as `TreeBuf::empty()` (length 0).
      - If true: assign the value column as length num_nodes (each node's staged
        value, 0 default where unset) and the `_present` column as length
        num_nodes (each node's staged present flag). This matches upstream where,
        once any node sets a stat, every node carries a (value, present) entry.
    Remove the current UNCONDITIONAL pushes of `data_count`, `data_count_present`,
    `sum_hess`, `sum_hess_present`, `gain`, `gain_present` from the per-node fill
    loop — they must no longer always be length num_nodes.

    Do NOT introduce fenced code; keep the existing `TreeBuf::from_owned` pattern.
    Leave `has_categorical_split` and `num_nodes` assignment as-is. Annotate each
    new column with a `tree.h:`-line reference comment (matching the file's
    existing convention of citing upstream line numbers).
