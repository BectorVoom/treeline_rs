## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → `Backend` selection (explicit) | The caller names the backend explicitly; there is no auto-detect/"best available" resolver (D-04). No network/auth/untrusted-input surface. |
| `*_case()` → GPU client construction | A missing device crosses back as the typed `DeviceUnavailable` from Plan 02 (or, per A3, a pre-construction probe avoids the FFI call entirely). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-05 | Denial of Service | `cuda_case()`/`wgpu_case()` constructing a client on the NVIDIA-less box | mitigate | Honor the Plan-01 A3 finding: if GPU `client()` hard-aborts, the `*_case()` pre-probes device availability and emits the skip without constructing; if it returns a catchable error, the Plan-02 `DeviceUnavailable` propagation suffices. Never auto-invoke an unavailable backend. |
| T-07-06 | Spoofing | "selected backend == backend that ran" provenance | mitigate | Each `*_case()` sets `backend: Backend::<X>` literally and routes only to that runtime; no fallback re-routing (D-04/D-05/D-09). |
| T-07-SC | Tampering | cargo installs | mitigate | No new crates; the harness forwards existing Plan-01 `treelite-cubecl` features only. Legitimacy posture from the RESEARCH audit holds. |

No high-severity threats: explicit backend registration over trusted in-process fixtures, no external-input surface. The realistic surface is the device-construction DoS (T-07-05), gated by the A3 finding.
